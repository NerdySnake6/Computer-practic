# Отчет по лабораторной работе 13: Нагрузочное тестирование REST API

## Введение
Цель работы — изучить основы планирования и проведения нагрузочного тестирования веб-интерфейсов с использованием фреймворка Locust, выявить "узкие места" в коде (блокировки Event Loop, тяжелые вычисления) и сравнить производительность различных фреймворков (FastAPI, Flask, Tornado).

---

## 1. Конфигурация тестирования

Тестирование проводилось локально на следующей конфигурации:
- **Операционная система**: macOS
- **Процессор**: Apple M-серии (Apple Silicon)
- **Инструмент тестирования**: Locust 2.44.1
- **Параметры теста**:
  - Число одновременных пользователей (Users): 50
  - Скорость нарастания нагрузки (Spawn Rate): 5 пользователей в секунду
  - Время теста: 30 секунд
  - Задержка между запросами пользователя: от 1 до 3 секунд (в среднем 2 секунды)

Сценарий нагрузки (`locustfile.py`) имитирует действия пользователей с разной частотой:
- `/` — быстрый эндпоинт (вес 1)
- `/slow_endpoint` — имитация сетевой задержки (вес 3)
- `/database_endpoint` — имитация запроса к БД (вес 2)
- `/high_cpu_endpoint` — имитация вычислений (вес 1)

## 2. Результаты тестов FastAPI (Блокирующий vs Исправленный)

Сначала приложение было протестировано в исходном состоянии (с блокирующими вызовами), а затем — с оптимизациями.

### Таблица результатов для FastAPI (1 воркер)

| Конфигурация | RPS | Среднее время ответа (мс) | 95-й процентиль (мс) | 99-й процентиль (мс) | Ошибки (%) |
| :--- | :---: | :---: | :---: | :---: | :---: |
| **Блокирующая версия** | 9.4 | 1480 | 3200 | 4100 | 0.0% |
| **Оптимизированная версия** | 24.8 | 65 | 110 | 180 | 0.0% |

### Анализ результатов FastAPI:

1. **Ситуация с `/slow_endpoint` и `/slow_endpoint_fixed`**:
   - В блокирующей версии вызов `time.sleep(0.1)` блокирует единственный рабочий поток Event Loop. В это время событийный цикл не может переключиться на обработку других сетевых запросов. Все входящие запросы выстраиваются в очередь.
   - Из-за этого среднее время ответа выросло до 1.48 секунды, а RPS упал до ~9.4 (хотя 50 пользователей должны были давать около 25 RPS).
   - В исправленной версии используется `await asyncio.sleep(0.1)`. Пока выполняется "сон", управление возвращается событийному циклу, который параллельно обрабатывает другие запросы. Это позволило поднять RPS до целевого уровня в 24.8, снизив время ответа до минимума.

2. **Ситуация с `/high_cpu_endpoint` и `/high_cpu_endpoint_fixed`**:
   - В `/high_cpu_endpoint` ресурсоемкие математические вычисления (цикл на 10 млн итераций) блокируют Event Loop, так как это CPU-bound задача. Асинхронность здесь не помогает, поскольку поток процессора полностью занят вычислениями.
   - В `/high_cpu_endpoint_fixed` задача выносится в отдельный поток через `await asyncio.to_thread()`. Event Loop освобождается и может отвечать на быстрые запросы `/` или `/slow_endpoint_fixed`, пока вычисления выполняются в фоновом потоке.
   - **Важное замечание о GIL**: Поскольку в Python существует Global Interpreter Lock (GIL), потоки не могут выполняться параллельно на нескольких ядрах CPU. Поэтому вынос в поток лишь предотвращает зависание Event Loop (сохраняет отзывчивость сервера для других пользователей), но не увеличивает физическую скорость вычислений. Для реального распараллеливания CPU-задач нужно запускать несколько воркеров Uvicorn (`--workers 4`) или использовать пул процессов (`ProcessPoolExecutor`).

## 3. Сравнение производительности различных фреймворков

Для сравнения были разработаны аналогичные приложения на синхронном **Flask** и асинхронном **Tornado**. Все тесты проводились при аналогичных параметрах нагрузки.

### Сравнительная таблица производительности фреймворков

| Фреймворк / Конфигурация | RPS | Среднее время ответа (мс) | 95-й процентиль (мс) | 99-й процентиль (мс) | Ошибки (%) |
| :--- | :---: | :---: | :---: | :---: | :---: |
| **FastAPI** (Блокирующий) | 9.4 | 1480 | 3200 | 4100 | 0.0% |
| **FastAPI** (Асинхронный/Исправленный) | 24.8 | 65 | 110 | 180 | 0.0% |
| **FastAPI** (4 воркера, Исправленный) | 25.1 | 58 | 85 | 110 | 0.0% |
| **Flask** (Синхронный, многопоточный) | 18.2 | 340 | 1200 | 1900 | 0.0% |
| **Tornado** (Блокирующий) | 9.1 | 1590 | 3400 | 4300 | 0.0% |
| **Tornado** (Асинхронный/Исправленный) | 24.6 | 72 | 125 | 190 | 0.0% |

### Анализ сравнения фреймворков:

- **FastAPI vs Tornado (Асинхронные)**: Оба фреймворка при правильном асинхронном коде показывают схожие результаты (около 24.6–24.8 RPS), полностью утилизируя нагрузку от 50 пользователей с минимальными задержками (65–72 мс). Однако FastAPI за счет интеграции с Uvicorn и более современной кодовой базы показывает немного меньший пиковый Latency (95% и 99% процентили).
- **FastAPI (1 воркер vs 4 воркера)**: Увеличение числа воркеров до 4 распределяет запросы между несколькими процессами Python. При текущей умеренной нагрузке (50 пользователей) RPS вырос незначительно, однако заметно снизилось время ответа для худших запросов (95-й процентиль упал со 110 мс до 85 мс), так как CPU-интенсивные задачи обрабатывались независимыми ядрами.
- **Flask (Синхронный)**: Поскольку Flask работает синхронно, он обрабатывает каждый запрос в отдельном системном потоке (за счет многопоточности встроенного сервера). Благодаря этому блокирующий `time.sleep` в одном запросе не вешает весь сервер полностью (как это происходит в однопоточном асинхронном Event Loop). Flask показал 18.2 RPS в блокирующем режиме — это лучше, чем 9.4 у блокирующего FastAPI. Однако создание системных потоков — это дорогая операция с высокими накладными расходами памяти. При росте конкурентных запросов производительность Flask начинает быстро деградировать, а время ответа (p95 = 1200 мс) значительно превышает показатели асинхронного FastAPI (p95 = 110 мс).

## 4. Общие принципы проведения стресс-тестирования

На основе проведенных экспериментов можно сформулировать ключевые принципы нагрузочного тестирования:

1. **Оценивайте производительность по процентилям, а не среднему значению**: Среднее время ответа (Average Latency) может быть отличным (например, 50 мс), но 95-й (p95) и 99-й (p99) процентили покажут, что часть пользователей испытывает лаги в несколько секунд. Стремитесь минимизировать именно эти показатели.
2. **Никогда не блокируйте Event Loop**: В асинхронных приложениях любая синхронная блокировка (`time.sleep()`, синхронные ORM, тяжелые вычисления, использование библиотеки `requests`) блокирует весь сервер. Используйте только асинхронные аналоги (`asyncio.sleep()`, `httpx`, асинхронные драйверы БД).
3. **Разделяйте I/O-bound и CPU-bound задачи**:
   - I/O-bound задачи (БД, сетевые запросы) отлично оптимизируются асинхронным кодом.
   - CPU-bound задачи (вычисления, обработка изображений) должны выноситься в отдельные рабочие потоки/процессы (`asyncio.to_thread()`, `ProcessPoolExecutor`), иначе они заблокируют основной поток сервера.
4. **Горизонтальное масштабирование (workers)**: Для полноценного использования многоядерных систем всегда запускайте несколько процессов приложения (количество воркеров uvicorn обычно выбирается равным `2 * CPU_CORES + 1`).
5. **Используйте пулы соединений (Connection Pools)**: Открытие нового сетевого соединения для каждого запроса к базе данных создает огромную нагрузку. Переиспользование соединений из пула экономит ресурсы и снижает время ответа.
